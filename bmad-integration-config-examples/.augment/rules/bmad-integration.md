# BMad-Method 集成规则

## 🎯 代码审查标准

### 架构合规性
- **模块化设计**: 确保代码遵循 BMad-Method 的模块化架构原则
- **依赖管理**: 检查依赖注入和模块间耦合度
- **接口设计**: 验证 API 接口设计符合 RESTful 或 GraphQL 标准
- **数据流**: 确保数据流向清晰，避免循环依赖

### 代码质量标准
- **命名规范**: 
  - 类名使用 PascalCase
  - 方法名使用 camelCase
  - 常量使用 UPPER_SNAKE_CASE
  - 文件名使用 kebab-case
- **注释要求**:
  - 公共 API 必须有 JSDoc 注释
  - 复杂业务逻辑必须有行内注释
  - 每个模块必须有文件头注释
- **错误处理**:
  - 所有异步操作必须有错误处理
  - 使用统一的错误处理机制
  - 错误信息必须有意义且可追踪

### 测试要求
- **单元测试覆盖率**: 新增代码覆盖率不低于 80%
- **集成测试**: 涉及多个模块的功能必须有集成测试
- **端到端测试**: 关键用户流程必须有 E2E 测试
- **测试命名**: 测试用例名称必须清晰描述测试场景

### 安全标准
- **输入验证**: 所有用户输入必须进行验证和清理
- **权限检查**: API 端点必须有适当的权限验证
- **敏感数据**: 不得在日志中记录敏感信息
- **依赖安全**: 检查第三方依赖的安全漏洞

### 性能要求
- **响应时间**: API 响应时间不超过 200ms
- **内存使用**: 避免内存泄漏和过度内存使用
- **数据库查询**: 避免 N+1 查询问题
- **缓存策略**: 合理使用缓存提升性能

## 🔧 BMad-Method 特定规则

### 工作流合规性
- **提交信息**: 遵循 BMad-Method 提交信息格式
  ```
  type(scope): description
  
  [optional body]
  
  [optional footer]
  ```
- **分支策略**: 遵循 GitFlow 分支管理策略
- **代码审查**: 所有代码必须经过至少一人审查

### 文档要求
- **README**: 每个模块必须有完整的 README 文档
- **API 文档**: 使用 OpenAPI/Swagger 规范
- **变更日志**: 维护 CHANGELOG.md 文件
- **架构文档**: 重要架构决策必须记录在 ADR 中

### 集成检查点
- **构建状态**: 确保 CI/CD 流水线通过
- **代码质量**: SonarQube 质量门禁通过
- **安全扫描**: 安全扫描无高危漏洞
- **依赖检查**: 依赖版本符合项目要求

## ⚠️ 严重问题标记

以下问题将被标记为严重问题，必须修复：

1. **安全漏洞**: SQL 注入、XSS、CSRF 等安全问题
2. **性能问题**: 明显的性能瓶颈或资源泄漏
3. **架构违规**: 违反 BMad-Method 架构原则的设计
4. **测试缺失**: 关键功能缺少测试覆盖
5. **文档缺失**: 公共 API 缺少文档
